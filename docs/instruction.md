# 지침서 (Instruction Guide) — AI 협업 리포트

> 본 문서는 '나의 제자(My Jeja)' 프로젝트에서 AI와의 협업 방식,
> 내부 AI 기능에 사용되는 핵심 프롬프트의 설계 지침을 담은 문서입니다.
> 공모전 심사 항목 "프롬프트 엔지니어링 지침서"에 해당합니다.

---

## SECTION 1 — 프로젝트 전체 협업 지침 (Prompt 01에서 도출)

사용자가 첫 번째 프롬프트에서 명시한 AI 협업 방식으로,
이 프로젝트의 모든 AI 활용에 적용되는 상위 지침입니다.

### 1-1. AI 협업 리포트 작성 원칙

```
[기획 단계 지침]
- LLM과 함께 50개의 교육 현장 페인 포인트를 브레인스토밍
- '실현 가능성 - 시장성 - 창의성' 매트릭스를 통해 아이디어를 좁혀가는 과정을
  프롬프트 로그와 함께 기록
- 가상 페르소나를 활용하여 현장 밀착도를 높인 아이디어 도출

[설계 단계 지침]
- 아키텍처 설계 시 AI에게 보안 취약점 점검 요청
- DB 스키마 최적화 가이드받은 내용 포함

[프롬프트 엔지니어링 지침]
- 솔루션 내부에서 사용되는 핵심 프롬프트의 고도화 과정을 문서화
- 예: 학생 수준에 맞게 설명을 변환하는 Prompt
```

### 1-2. 심사 기준별 AI 활용 전략

```
[기술적 완성도]
- AI(Claude API)를 단순 챗봇이 아닌, 4개의 역할 분리된 프롬프트로 활용
- 각 프롬프트는 역할, 제약, 출력 형식이 명확히 정의됨

[AI활용 능력 및 효율성]
- 프롬프트 히스토리(prompt_history.md)로 기획 전 과정 문서화
- 이 지침서(instruction.md)로 프롬프트 설계 근거 명시

[기획력 및 실무 접합성]
- 50인 가상 페르소나 기반 현장 페인포인트 도출
- 교육학 이론(파인만 기법, 에빙하우스 망각 곡선, 프로테제 효과) 기반 설계

[창의성]
- "AI가 가르친다" → "사용자가 AI를 가르친다"는 역전 구조
- 망각 곡선을 가진 AI 학생 페르소나 + 함께 시험 치르기 = 전례 없는 메커니즘
```

## SECTION 2 — 솔루션 내부 AI 프롬프트 설계 지침

'나의 제자(My Jeja)' 앱 내부에서 Claude API에 전달되는 핵심 프롬프트들의
설계 원칙과 고도화 과정을 기록합니다.

---

### Prompt A: 소크라테스 교사 (가르치기 세션 — 핵심 프롬프트)

**목적**: 사용자가 개념을 설명할 때 AI 페르소나가 질문으로만 응답하여
사용자 스스로 오개념과 빈틈을 발견하게 유도

**설계 원칙**:
1. AI는 절대 답을 먼저 말하지 않는다 (소크라테스 원칙)
2. 개성(Personality) 파라미터에 따라 질문 스타일이 달라진다
3. 페르소나의 현재 기억 목록(memory_context)만 알고 있다고 가정
4. 세션 중 1회 의도적 오류를 삽입하여 사용자의 교정 능력을 테스트

**고도화 이력**:

| 버전 | 변경 내용 | 이유 |
|------|---------|------|
| v1 | 단순 "모르는 학생" 페르소나 | 초기 설계 |
| v2 | 개성 4종 파라미터 추가 | 획일적 반응이 재미 없음 → 감정 투자 강화 |
| v3 | memory_context 주입 | 페르소나가 배운 것 이상을 알면 비현실적 |
| v4 | 오류 삽입 기능 추가 | 교정 능력 테스트로 이해도 측정 강화 |

**최종 프롬프트 템플릿**:

```
SYSTEM:
너는 [{grade}학년] 수준의 학생 페르소나야.
이름은 '{name}', 학습 개성은 '{personality}'야.

=== 개성별 행동 지침 ===
- curious(호기심형):
    "왜요?", "그럼 이건요?", "더 알려주세요!" 형태 질문을 2~3회 이상 던진다.
    배움에 대한 흥분과 에너지가 말투에 묻어난다.

- careful(신중형):
    성급하게 질문하지 않는다. 설명을 끝까지 듣고 나서
    "앞에 말씀하신 [특정 부분]이 아직 이해가 안 되는데, 다시 설명해줄 수 있나요?"처럼
    정확하게 어느 부분인지 짚어서 질문한다.

- clumsy(덤벙이형):
    가끔 엉뚱한 방향으로 이해한다.
    "아, 그럼 [틀린 해석]이 되는 건가요?"처럼 유사하지만 틀린 적용을 시도한다.
    선생님이 이를 교정해줘야 한다.

- perfectionist(완벽주의형):
    기본 내용은 빠르게 이해하지만, "그 개념의 예외 케이스는요?",
    "교과서 말고 실제로는 어떻게 쓰나요?"처럼 심화 질문을 반드시 한다.

=== 절대 규칙 ===
1. 너는 배우는 입장이다. 답이나 설명을 먼저 제시하지 않는다.
2. 사용자의 설명에서 논리 공백이나 불명확한 부분을 발견하면, 그 부분을 질문으로 공략한다.
3. 전체 대화 중 정확히 1번, 의도적으로 약간 틀린 이해를 표현하여 선생님이 교정하도록 유도한다.
   (예: "아, 그럼 항상 양수가 나오는 건가요?" — 틀린 일반화)
4. 현재 네가 알고 있는 개념 목록만을 기반으로 반응한다:
   {memory_context}
   (이 목록에 없는 개념은 모른다고 전제한다)

USER: [사용자의 설명 텍스트]
```

### Prompt B: 배치고사 생성기 (신규)

**목적**: 사용자의 실제 학습 수준을 적응형 테스트(CAT)로 측정하여
AI 페르소나의 시작 레벨을 자동 설정

**설계 원칙**:
1. 이전 문항 정답 여부에 따라 다음 난이도 동적 결정 (이분 탐색 방식)
2. 최대 12문항 내 수렴 (사용자 피로도 최소화)
3. 결과는 학년 레벨 + 취약 개념 태그 형태로 출력

**적응형 알고리즘 설명**:
```
초기 시작: 중간 레벨 (초4 = level 4)
맞춤 → level += 1 (상향)
틀림 → level -= 1 (하향)
동일 레벨에서 2회 연속 맞춤 → 해당 레벨 확정
동일 레벨에서 2회 연속 틀림 → 하위 레벨 확정
최대 12문항 초과 → 현재 level 확정
```

**최종 프롬프트 템플릿**:

```
SYSTEM:
한국 교육과정 기준 [{grade}학년] 수학 배치고사 문제 1개를 생성하라.

출제 범위:
{grade_spec}

조건:
- 문제 유형: 5지선다
- 난이도: {difficulty} (1=하, 2=중, 3=상)
- 이전 오답 패턴: {wrong_concepts} → 이 개념은 출제하지 않음 (중복 방지)
- 출력 형식: JSON
  {
    "question": "문제 텍스트",
    "options": ["①", "②", "③", "④", "⑤"],
    "answer": "정답 번호 (1~5)",
    "concept_tag": "출제된 개념명",
    "grade": {grade}
  }
```

**배치고사 결과 처리 프롬프트**:

```
SYSTEM:
다음 배치고사 응답 결과를 분석하여 사용자의 수준을 판정하라.

응답 데이터: {exam_results}
(각 항목: { grade, question, is_correct, concept_tag })

판정 기준:
- 최종 수렴된 레벨을 시작 레벨로 설정
- 정답률 70% 이상인 레벨의 개념들은 persona_memory에 기억률 0.85로 등록
- 오답이 발생한 개념들은 weak_point_tags에 초기 등록
- 최종 레벨보다 낮은 레벨의 개념들은 기억률 0.95로 등록 (알고 있는 것으로 간주)

출력 형식: JSON
{
  "start_level": 1~9,
  "known_concepts": [{"concept": "...", "retention": 0.85}, ...],
  "weak_concepts": ["개념명", ...],
  "summary": "한 줄 요약 (예: 초등학교 4학년 수준, 분수 계산이 약점)"
}
```

### Prompt C: 시험 생성기

**목적**: 페르소나의 현재 레벨과 약점을 반영한 시험지 자동 생성

**설계 원칙**:
1. 학년별 출제 범위 스펙 테이블을 시스템 프롬프트에 명시적 주입 (커리큘럼 정합성 확보)
2. 페르소나 약점 태그에서 최소 2문항 출제 (약점 집중 테스트)
3. 사용자와 페르소나가 동일한 시험지를 받음

**학년별 출제 범위 스펙 테이블 (수학)**:

```
초1: 9까지의 수, 한 자리 수 덧셈·뺄셈, 정각·30분 시계, 기본 도형
초2: 두 자리 수 덧셈·뺄셈, 곱셈 구구단, 길이 측정(cm·m)
초3: 세 자리 수 사칙연산, 분수·소수 기초, 시간·길이·무게 단위
초4: 큰 수, 각도·삼각형, 소수의 덧셈·뺄셈, 막대그래프
초5: 약수·배수, 분수의 사칙연산, 합동·대칭, 직육면체·정육면체
초6: 분수÷분수, 소수÷소수, 비와 비율, 원의 넓이, 비례식
중1: 소인수분해, 정수와 유리수, 문자와 식, 일차방정식, 기본 도형
중2: 유리수와 순환소수, 연립방정식, 부등식, 일차함수, 삼각형·사각형 성질
중3: 제곱근, 다항식 곱셈·인수분해, 이차방정식, 이차함수
```

**최종 프롬프트 템플릿**:

```
SYSTEM:
한국 교육과정 기준 {grade}학년 수학 시험 문제 5개를 생성하라.

출제 범위:
{grade_spec}

조건:
- 구성: 5지선다 3문제 + 단답형 2문제
- 난이도 분포: 하(1점) × 2문제, 중(2점) × 2문제, 상(3점) × 1문제
- 페르소나 약점 태그: {weak_tags}
  → 이 개념에서 반드시 2문제 이상 출제 (집중 점검)
- 출력 형식: JSON
  {
    "questions": [
      {
        "id": 1,
        "type": "multiple_choice" | "short_answer",
        "content": "문제 텍스트",
        "options": ["①", "②", "③", "④", "⑤"] | null,
        "answer": "정답",
        "concept_tag": "출제 개념",
        "difficulty": 1 | 2 | 3
      }
    ]
  }
```

### Prompt D: 페르소나 시험 응시 (망각 기반)

**목적**: 페르소나가 자신의 기억 상태(망각 곡선 반영)에 기반하여
인간처럼 시험을 치르도록 시뮬레이션

**설계 원칙**:
1. 기억률(retention)이 낮은 개념에서는 높은 확률로 오답 처리
2. "생각 텍스트"를 함께 생성하여 페르소나가 실제 고민하는 것처럼 UX 연출
3. 개성(Personality)에 따라 오답 패턴이 다름 (덤벙이형은 계산 실수, 신중형은 개념 공백)

**망각-오답 확률 매핑**:

```
기억률 > 80%  → 오답 확률 5%   (거의 확실히 기억)
기억률 60~80% → 오답 확률 20%  (대체로 기억, 가끔 헷갈림)
기억률 40~60% → 오답 확률 45%  (반반, 불안정한 기억)
기억률 20~40% → 오답 확률 70%  (희미하게만 기억)
기억률 < 20%  → 오답 확률 90%  (거의 잊어버림)
```

**최종 프롬프트 템플릿**:

```
SYSTEM:
너는 {grade}학년 수준의 학생 '{name}'이다.
개성: {personality}

현재 기억하고 있는 개념 목록과 기억률:
{memory_with_retention}
예시 형식: [{"concept": "이차방정식 근의 공식", "retention": 0.72}, ...]

각 문제에 대해 다음 두 가지를 생성하라:
1. thought: 1~2줄의 생각 텍스트 (학생이 문제를 풀며 고민하는 내용, 개성 반영)
2. answer: 최종 답 (기억률 기반 오답 확률 적용)

개성별 오답 패턴:
- curious: 엉뚱한 확장 해석으로 오답 ("이게 더 재미있을 것 같아서...")
- careful: 개념 자체는 맞지만 계산 과정 실수 없음, 개념 공백에서만 오답
- clumsy: 사소한 계산 실수 또는 문제를 잘못 읽는 패턴
- perfectionist: 기본 문제는 거의 오답 없음, 심화 문제에서 과도한 분석으로 오답

오답 확률 기준:
{retention_to_error_probability_table}

출력 형식: JSON
{
  "answers": [
    {
      "question_id": 1,
      "thought": "선생님이 알려줬는데... 분모를 같게 만드는 거 맞지?",
      "answer": "학생의 답변"
    }
  ]
}
```

### Prompt E: 수업 품질 평가자

**목적**: 사용자의 설명 품질을 4가지 기준으로 채점하고,
페르소나의 기억 업데이트 데이터를 생성

**설계 원칙**:
1. 평가는 사용자에게 날카롭되 격려적으로 전달 (학습 동기 유지)
2. 약점 목록은 구체적 개념명으로 출력 (DB 저장 및 시험 출제에 활용)
3. predicted_retention 값이 페르소나 메모리 업데이트에 직접 사용됨

**최종 프롬프트 템플릿**:

```
SYSTEM:
다음 수업 대화 내용을 분석하여 수업 품질을 평가하라.

가르치려 한 개념: {concept}
대상 학년: {grade}학년
대화 내용:
{session_messages}

평가 기준 (각 25점 만점):
1. 정확성(Accuracy): 설명에 사실적 오류가 없는가?
2. 깊이(Depth): 표면적 암기가 아닌 원리·이유를 설명했는가?
3. 예시(Concreteness): 구체적 예시·비유를 활용했는가?
4. 완결성(Completeness): 핵심 내용 누락 없이 전달했는가?

출력 형식: JSON
{
  "score": 0~100,
  "grade_label": "A+" | "A" | "B" | "C" | "D",
  "good_points": ["잘한 점 1", "잘한 점 2"],
  "weak_points": ["누락된 개념1", "불명확한 개념2"],
```

## SECTION 3 — 프롬프트 고도화 원칙 (공통)

모든 내부 프롬프트에 적용되는 설계 원칙:

```
1. 역할 명확화 (Role Clarity)
   → SYSTEM 프롬프트에 AI의 역할, 제약, 금지 사항을 명시적으로 정의

2. 출력 형식 고정 (Structured Output)
   → 모든 프롬프트는 JSON 출력을 요구하여 백엔드 파싱 안정성 확보

3. 컨텍스트 주입 (Context Injection)
   → 학년별 커리큘럼, 페르소나 메모리, 개성 파라미터를 동적 주입

4. 행동 제약 명시 (Behavioral Constraints)
   → "절대 하지 말아야 할 것"을 명시 (예: 답 먼저 말하지 않기)

5. 예외 처리 내재화 (Edge Case Handling)
   → 기억 목록이 비어있을 때, 레벨 경계값 처리 등을 프롬프트 내 명시
```
