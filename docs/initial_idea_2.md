## STEP 3 — 창의성 관점 재브레인스토밍: 발상의 전환 6가지

`initial_idea.md`의 15개 후보에서 벗어나, **"교육의 당연한 방향을 뒤집는"** 아이디어를 별도로 도출.

### [역발상 축: AI가 가르치는 게 아니라, 학생이 AI를 가르친다]

| 발상 | 핵심 역전 | 교육학 근거 |
|------|---------|-----------|
| 학생 → AI 설명 | "AI가 모른다"는 설정으로 학생이 교사 역할 | 파인만 기법(Feynman Technique), 프로테제 효과(Protégé Effect) |
| AI가 일부러 틀린 개념 제시 → 학생이 교정 | 오류 발견이 이해를 깊게 함 | 오류 교정 학습(Error Correction Learning) |

> **교육 피라미드 (NTL Institute)**: 강의 듣기 5% 기억 → **남에게 가르치기 90% 기억**
> 현재 모든 AI 교육 앱은 "듣기/읽기" 영역에만 존재. **가르치기 영역은 공백.**

### [역발상 축: 학습 결과가 아닌 학습 '과정의 감정'을 측정]

| 발상 | 핵심 역전 |
|------|---------|
| 점수가 아니라 '이해되는 순간(AHA Moment)'을 기록 | 학습의 질을 감정 변화로 측정 |
| 번아웃 징후를 패턴으로 감지해 학습 방식 변경 제안 | 학습량이 아닌 학습 컨디션 최적화 |

### [역발상 축: 교사가 콘텐츠를 만들지 않고, AI가 교사의 말투를 학습]

| 발상 | 핵심 역전 |
|------|---------|
| 교사의 설명 패턴을 학습해 교사 없이도 동일한 피드백 | 교사 1인의 영향력을 무한 복제 |

## STEP 5 — 🥇 최종 선정: **"가르쳐봐 (TeachBack AI)"**

> *"AI에게 설명하다 보면, 내가 뭘 모르는지 스스로 깨닫는다"*

### 핵심 개념: 학습의 방향을 역전시킨다

```
기존 모든 AI 교육앱:   AI ──설명──▶ 학생
TeachBack AI:        학생 ──설명──▶ AI (소크라테스 역할)
```

학생이 오늘 공부한 개념을 AI에게 설명한다. AI는 "그게 왜 그런가요?", "이 부분은 이해가 안 되네요"라며 질문을 던진다. 학생의 설명 속 **오개념, 논리 공백, 피상적 암기**를 AI가 감지하고, 직접 답을 주는 대신 학생 스스로 찾도록 유도한다.

세션이 끝나면: **"이해도 점수 + 놓친 핵심 개념 3가지 + 다음에 집중할 포인트"** 제공.

### 왜 창의적인가? — 국내외 시장 공백 분석

| 유형 | 기존 솔루션 | TeachBack의 차이 |
|------|-----------|----------------|
| AI 챗봇 튜터 | 뤼이드, 콴다, Khanmigo | AI가 설명 → 수동적 학습자 |
| 플래시카드 | Anki, Quizlet | 암기 반복 → 표면적 이해 |
| AI 문제풀이 | 매스프레소, 오투 | 정답 여부만 판단 |
| **TeachBack** | **국내외 유사 제품 없음** | **학생이 능동적 교사 역할** |

> **검증된 교육학 이론 3가지에 동시 기반**:
> 1. **파인만 기법**: 개념을 쉽게 설명할 수 있어야 진짜 아는 것
> 2. **프로테제 효과(Protégé Effect)**: 누군가를 가르친다고 생각할 때 학습 효율 극대화
> 3. **소크라테스 문답법**: 질문을 통한 자기 발견이 강의보다 깊은 이해 형성

```

- AI가 일부러 틀린 유도 질문 1회 삽입 (학생 교정 능력 테스트)

[세션 결과]
- 이해도 점수 (0~100점) + 한 줄 평가
- 놓친 핵심 개념 최대 3개 하이라이트
- "다음에 집중할 포인트" 1문장 추천

[학습 히스토리]
- 날짜별 세션 기록
- 과목별 이해도 변화 그래프 (간단한 선 그래프)
- 반복적으로 틀리는 개념 태그 누적 표시
```

```

- 음성 입력/TTS 출력

```

### 핵심 프롬프트 설계 초안 (Prompt Engineering 지침서 예고)

TeachBack의 품질은 **AI의 질문 품질**에 달려 있다. 3개 역할 프롬프트를 분리 설계:

#### 역할 1: 소크라테스 질문자 (기본 모드)
```
시스템 프롬프트 방향:
- AI는 해당 과목의 개념을 '잘 모르는 호기심 많은 학습자' 페르소나
- 학생의 설명을 듣고 "왜요?", "그게 어떻게 되나요?", "예시를 들어줄 수 있나요?" 형태로만 응답
- 절대 직접 답변하지 않음 (답 금지 원칙)
- 학생 설명의 논리적 빈틈을 발견하면 그 빈틈을 질문으로 공략
```

#### 역할 2: 오류 삽입자 (도발 모드, 세션 중 1회)
```
시스템 프롬프트 방향:
- 대화 중반 1회, AI가 의도적으로 틀린 개념으로 반박
- 예: "아, 그럼 이차방정식은 항상 실수 해가 2개 있는 거 아닌가요?"
- 학생이 이 오류를 교정하면 +보너스 점수
- 학생이 동조하면 오개념 태그에 기록
```

#### 역할 3: 평가자 (세션 종료 후)
```
시스템 프롬프트 방향:
- 전체 대화 내용을 입력받아 JSON 형식으로 평가 출력
- 출력: { score: 0~100, missed_concepts: [], next_focus: "", encouragement: "" }
- 점수 기준: 정확성(40%), 설명 깊이(30%), 예시 활용(20%), 오류 교정 능력(10%)
```

## AI 협업 리포트 메모 (기획 2단계)

**조건 변경에 따른 재평가 과정**:

기존 D+F 통합안이 제약 조건(1주 개발, 창의성 우선)에 부합하지 않음을 CEO·CTO 시각으로 재검토.
핵심 재발견: "AI가 가르친다"는 패러다임 자체가 모든 기존 에듀테크의 공통 전제이며, 이 전제를 뒤집는 것이 가장 강력한 차별화 포인트임을 확인.

**참조한 교육학 이론**:
- Feynman Technique (파인만, 1960s) — 개념을 초등학생에게 설명할 수 있을 때 진짜 이해한 것
- Protégé Effect (Palinscar & Brown, 1984) — 학습자가 교사 역할을 맡을 때 자신의 학습 동기와 깊이 모두 상승
- Socratic Method — 답을 주는 것보다 질문이 더 깊은 이해를 만든다

**다음 단계**: 설계 단계 문서 작성 (아키텍처 다이어그램 + 보안 취약점 점검 + DB 스키마 최적화)
